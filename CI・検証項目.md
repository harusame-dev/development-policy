# CI（検証項目）

## 概要

CI で標準的に実行する検証項目を定義する。新規プロジェクトはこの一覧を基準に CI を構成し、いずれかの項目を省く場合はワークフローにその理由をコメントで残す。

「どの項目を検証するか」を方針として定め、項目同士の実行順序・並列構成・マージ必須範囲を設計として示す。ワークフロー自体の記述ルール（パーミッション・タイムアウト・SHA ピン留め・不要実行のスキップなど）は [CI（GitHub Actions）](./CI・githubactions.md) を参照する。

## 1. 標準検証項目

### 方針

CI では、以下の検証項目をすべて実行することを標準とする。

| #   | 項目                             | 検証内容                                                       |
| --- | -------------------------------- | ------------------------------------------------------------ |
| 1   | フォーマットチェック             | コード整形が規約どおりか                                       |
| 2   | 静的解析（Lint）                 | バグの温床・アンチパターン・規約違反                           |
| 3   | 型チェック                       | 型の整合性（コンパイルを通さず型のみ検証）                     |
| 4   | デッドコード検出                 | 未使用の export・依存・ファイル                               |
| 5   | ユニット・インテグレーションテスト | ロジック・モジュール結合の振る舞い                            |
| 6   | UI テスト                        | コンポーネントの描画・操作に対する振る舞い                     |
| 7   | テストカバレッジチェック         | E2E 以外のテストのカバレッジが基準を満たすか                   |
| 8   | E2E テスト                       | ユーザー操作シナリオが end-to-end で成立するか                 |
| 9   | ビルドチェック                   | 本番ビルドが成功するか                                         |
| 10  | アクセシビリティチェック         | 支援技術で利用可能な UI になっているか                         |

各項目で採用するツールは [ライブラリ](./ライブラリ.md) を参照する。

### 理由

- **品質ゲートを共通言語化する**: どのプロジェクトでも「CI で守る最低ライン」が同じになり、レビューや移管のたびに検証範囲を議論し直さなくて済む
- **不具合を作り込んだ層で止める**: フォーマット崩れから本番ビルド失敗・アクセシビリティ違反まで、種類の異なる欠陥をそれぞれ専用の検証で早期に検知できる
- **省略を明示的にする**: 標準を定めることで「やっていない」ことが可視化され、項目を外す判断には必ず理由が伴うようになる

## 2. 各検証項目

各項目の「何を検証するか」と「なぜ標準に含めるか」を示す。実装の詳細はリンク先の各ドキュメントに従う。

### 2.1 フォーマットチェック

コードの整形がフォーマッタの規約どおりかを検証する（自動整形ではなく差分の有無のチェック）。整形は機械的に一意化できる領域であり、レビューで指摘するのは時間の無駄であるため、CI で機械的に弾く。

### 2.2 静的解析（Lint）

バグの温床になりやすい記述・アンチパターン・コーディング規約違反を検出する。型チェックでは捕まえられない「型は正しいが危険なコード」（未使用変数、到達不能分岐、危険な依存配列など）をカバーする。

### 2.3 型チェック

`tsc --noEmit` などで型の整合性のみを検証する。エディタ上では見落とされた型エラーや、ビルド対象外ファイルの型崩れを CI で確実に検知する。ビルドチェックとは独立させ、ビルド設定（トランスパイラ）が型エラーを無視する構成でも型崩れを見逃さないようにする。

### 2.4 デッドコード検出

未使用の export・未使用の依存パッケージ・到達不能なファイルを検出する。放置されたコードは読み手の認知負荷とメンテナンスコストを増やすため、増え続ける前に CI で検知する。

### 2.5 ユニット・インテグレーションテスト

ロジックおよびモジュール結合の振る舞いを検証する。テストの粒度・構造・記法・モックの扱いは [テスト](./テスト.md) に従う。CI ではテストサイズに応じて並列度・タイムアウトを調整する。

### 2.6 UI テスト

コンポーネントの描画・操作に対する振る舞いを検証する。要素の取得はアクセシビリティクエリを用い（[テスト](./テスト.md) 参照）、実装詳細に依存したセレクタは使用しない。

### 2.7 テストカバレッジチェック

E2E 以外のテストのカバレッジが基準を満たすかを検証する。測定対象・達成基準（ブランチカバレッジ 80% 以上）は [テスト（カバレッジ）](./テスト・カバレッジ.md) に従う。基準を下回った場合は CI を失敗させる。

### 2.8 E2E テスト

ユーザーが達成したい目的を end-to-end で実行できるかを検証する。実行コスト・安定性の面で重いため、対象は Medium で代替できないシナリオに絞る（[テスト](./テスト.md) 参照）。カバレッジ測定の対象外とする。

### 2.9 ビルドチェック

本番ビルドが成功することを検証する。型チェックやテストが通っても、ビルド固有の設定（環境変数・バンドル・静的解析・コード生成）でのみ顕在化する破壊を検知する。

### 2.10 アクセシビリティチェック

支援技術で利用可能な UI になっているかを検証する。実行時の自動チェック（アクセシビリティ検証を E2E に組み込む）を行う。ルールは [アクセシビリティ](./アクセシビリティ.md)、ツールは [ライブラリ](./ライブラリ.md) に従う。

## 3. マージ必須条件

### 方針

標準検証項目はすべてマージの必須条件（required status checks）に指定する。

### 理由

- **品質ゲートを実効化する**: 必須条件にしなければ、失敗した検証を見落としたままマージでき、CI を設けた意味が失われる。必須指定によって「全項目グリーンでなければマージできない」状態を保証する

### 設計

- マージ必須に指定したジョブは、変更内容に関係なく必ず完了させる必要がある。関連ファイルが変更されていない場合のスキップは、ジョブ自体を起動しない `paths-ignore` ではなく、ステップレベルで制御する（[CI（GitHub Actions）](./CI・githubactions.md) の「不要なワークフローのスキップ」を参照）
