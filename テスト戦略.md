# テスト戦略

## 概要

自動テストの粒度・構造・実装スタイルを統一し、信頼性と保守性の高いテストスイートを構築するための戦略。

## 1. テストサイズの分類

### 方針

Google のテストサイズ ([Small / Medium / Large](https://testing.googleblog.com/2010/12/test-sizes.html)) の考え方を採用し、各テストがどのサイズに該当するかを意識して設計・実装する。

| サイズ   | 実行範囲                                       | 外部依存                                                                  | 実行時間の目安 |
| -------- | ---------------------------------------------- | ------------------------------------------------------------------------- | -------------- |
| Small    | 単一プロセス・単一スレッド内で完結             | 不可（ネットワーク・ファイル I/O・DB・スリープ・マルチスレッドはすべて不可） | 数十ミリ秒以内 |
| Medium   | 単一マシン内で完結                             | localhost に閉じたネットワーク、ファイル I/O、ローカル DB は可            | 数秒以内       |
| Large    | 複数マシン・外部システムを含む                 | 外部 API、外部 DB、ブラウザ越しの E2E など制約なし                        | 数秒〜数分     |

### 理由

- テストサイズを共通言語にすることで、各テストの依存範囲・実行コスト・期待される安定性をチーム全員が同じ尺度で議論できる
- サイズごとに実行頻度・並列度・タイムアウト設定を分けられ、CI のフィードバックループを最適化できる
- 「単体テスト」「結合テスト」のような曖昧な分類より、依存範囲という客観的な基準で分けられるため、判断のブレが少ない

## 2. テストサイズの優先順位

### 方針

Medium サイズのテストをベースとして優先的に実装する。Small は補助的に、Large は最小限に留める。

### 理由

- **Medium を主軸にする理由**: 実プロダクトに近い構成（ローカル DB やローカルサーバーを含む）で検証できるため、結合バグやスキーマ不整合などプロダクトの破壊につながりやすい不具合を高い確度で検知できる。Small だけでは過剰なモックに依存し、リファクタ耐性が低く偽陽性・偽陰性が増える
- **Small を補助にする理由**: 純粋ロジック（バリデーション、計算、変換）は Small で網羅した方が高速かつ安定する。ただし Small は「依存を全てモックした結果、本番経路を通っていない」テストになりやすいため、振る舞いの保証ではなくロジックの網羅に用途を限定する
- **Large を最小限にする理由**: 実行時間・安定性・メンテナンスコストすべてで不利なため、ユーザー操作シナリオなど Medium で代替不能な範囲に絞る。Large でしか守れない契約以外は Medium に押し下げる

### 実装方法

- 新規にテストを書くときは、まず Medium で書けないかを検討する
- 純粋関数や複雑な分岐を持つロジックのみ Small で網羅する
- 認証フロー・決済フローなど、ユーザー視点での通しの検証が必要な箇所のみ Large（E2E）でカバーする
- CI では Small → Medium → Large の順で並列実行し、早期にフィードバックを返す

## 3. テストの構造

### 方針

`describe` などのネスト構造は使用せず、フラットな構造でテストを記述する。

### 理由

- **テスト名だけで意図が伝わる**: フラット構造ではテスト名に状況と期待結果を完結に書く必要があるため、テスト単体で意味が読み取れる。ネスト構造ではタイトルが `describe` と `test` に分断され、失敗ログだけでは文脈の再構築にコストがかかる
- **共有状態の混入を防げる**: `describe` は `beforeEach` / `afterEach` などライフサイクルフックと組み合わさってネスト固有の暗黙の前提を生みやすい。フラット構造はそのような共有状態の入り込む余地がなく、テストごとの独立性が高まる
- **テスト一覧の並び替え・抽出が容易**: 階層に縛られず、テスト名の命名規約だけでフィルタや並び替えができる

### 実装方法

- ファイル単位で対象（関数・コンポーネント・エンドポイントなど）を 1 つに絞る
- テスト名は「対象 + 条件 + 期待結果」が分かる文を 1 行で書く
- 階層で表現したくなる関心事は、ファイル分割で表現する

```ts
// NG: describe でネストする
describe("createUser", () => {
  describe("when email is invalid", () => {
    test("returns ValidationError", () => {});
  });
});

// OK: フラット構造でテスト名に状況と期待結果を込める
test("createUser はメールアドレスが不正な場合 ValidationError を返す", () => {});
test("createUser はメールアドレスが重複している場合 ConflictError を返す", () => {});
```

## 4. 事前準備・後処理の実装方法

### 方針

`beforeEach` / `afterEach` / `beforeAll` / `afterAll` は使用せず、`test.extend` によるフィクスチャ形式で実装する。事前準備（データ作成・依存リソースの起動）と後処理（データ削除・リソースの解放）はフィクスチャ側に閉じ込め、テスト本体は検証対象の振る舞いの記述に集中させる。

### 理由

- **テストごとに必要な前提だけを宣言できる**: フィクスチャは引数として受け取った場合のみセットアップが走るため、テストごとに必要なものを宣言的に指定できる。`beforeEach` のように「ファイル内のすべてのテストに同じ前提が暗黙に適用される」状態を避けられる
- **後処理の漏れがなくなる**: フィクスチャ内で `use` の前後にセットアップとクリーンアップを定義する構造により、テストが失敗・スキップされてもクリーンアップが必ず実行される。`afterEach` の書き忘れ・対象漏れによるリソースリークを防げる
- **テスト本体の見通しが良くなる**: テストファイル冒頭にライフサイクルフックが並ばないため、テスト本体には「何を検証しているか」だけが残り、第三者が読んだときの認知負荷が下がる
- **再利用と組み合わせが容易**: フィクスチャは関数として export でき、複数のテストファイル間で再利用できる。`beforeEach` のように特定のファイル / スコープに紐付かない

### 実装方法

#### Vitest の場合

```ts
import { test as base, expect } from "vitest";

type Fixtures = {
  user: { id: string; email: string };
  db: Database;
};

const test = base.extend<Fixtures>({
  db: async ({}, use) => {
    const db = await openTestDatabase();
    await use(db);
    await db.close();
  },
  user: async ({ db }, use) => {
    const user = await db.users.create({ email: "test@example.com" });
    await use(user);
    await db.users.delete(user.id);
  },
});

test("getUser は登録済みユーザーを取得できる", async ({ user }) => {
  const result = await getUser(user.id);
  expect(result.email).toBe("test@example.com");
});

test("getUser は存在しない ID の場合 null を返す", async ({ db }) => {
  const result = await getUser("non-existent-id");
  expect(result).toBeNull();
});
```

#### Playwright の場合

```ts
import { test as base, expect } from "@playwright/test";

type Fixtures = {
  loggedInPage: Page;
};

const test = base.extend<Fixtures>({
  loggedInPage: async ({ page }, use) => {
    await page.goto("/login");
    await page.fill("[name=email]", "test@example.com");
    await page.fill("[name=password]", "password");
    await page.click("button[type=submit]");
    await use(page);
    // セッションのクリーンアップなど
  },
});

test("ダッシュボードはログイン済みユーザー名を表示する", async ({ loggedInPage }) => {
  await expect(loggedInPage.locator("[data-testid=user-name]")).toHaveText("test@example.com");
});
```

#### 設計上の注意

- フィクスチャは「テストが必要とする前提」を表現するものとし、テスト内で実施する操作（クリック、API 呼び出しの検証など）は含めない
- 1 つのフィクスチャに複数の責務を詰め込まず、フィクスチャ間の依存（`async ({ db }, use) => ...`）で組み立てる
- 共有可能なフィクスチャは `fixtures/` ディレクトリ等に切り出し、複数のテストファイルから import して再利用する
