# アクセシビリティ

Web フロントエンドにおけるアクセシビリティのルールをまとめる。

## 1. テキスト代替には `sr-only` を優先する

### 方針

アイコンのみのボタンなど、視覚的にラベルが無い要素のテキスト代替には、`aria-label` ではなく `sr-only` クラスを付けたテキスト要素を優先する。

### 理由

- **機械翻訳が効く**: `aria-label` の値は機械翻訳の対象から外れやすく、ページを翻訳しても元の言語のまま読み上げられることがある。`sr-only` は通常の DOM テキストなので翻訳が効く
- **崩れに気づきやすい**: `aria-label` は画面に表示されないため、文言の不整合や付け忘れがレビューや動作確認で見落とされやすい。`sr-only` のテキストは DOM 上に実在するため、検証やテストで参照しやすい

### 実装方法

```tsx
// 非推奨: aria-label
<button aria-label="検索">
  <SearchIcon />
</button>

// 推奨: sr-only クラスを付けたテキスト要素
<button>
  <SearchIcon />
  <span className="sr-only">検索</span>
</button>
```

## 2. 振る舞いと見た目の一致

### 方針

要素の振る舞いと見た目を一致させる。

- ページ遷移を行う場合は `a` / `Link` を使い、見た目もリンク（テキストリンクなど）にする
- 状態変更や副作用のある操作は `button` を使い、見た目もボタンにする
- 「リンクをボタンの見た目にする」「ボタンをテキストリンクの見た目にする」といった、振る舞いと外観の不一致は避ける

### 理由

- **操作結果が予測できる**: 見た目から振る舞いが推測できることで、ユーザーは操作前に結果（遷移するのか、その場で処理が走るのか）を予測できる
- **要素のセマンティクスを正しく保てる**: 遷移は `a` / `Link`、操作は `button` という対応を守ることで、キーボード操作・スクリーンリーダーの読み上げ・右クリックメニューなど、各要素が本来持つ挙動が正しく働く

### 実装方法

```tsx
// 非推奨: 遷移なのにボタン、操作なのにリンク
<button onClick={() => router.push("/detail")}>詳細を見る</button>
<a href="#" onClick={submit}>送信</a>

// 推奨: 遷移はリンク、操作はボタン（見た目もそれぞれに合わせる）
<Link href="/detail">詳細を見る</Link>
<button onClick={submit}>送信</button>
```
