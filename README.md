# development-policy

開発の方針・規約を集約するリポジトリ。
各ドキュメントは粒度に応じて **ポリシー / 設計 / ルール** の3カテゴリのいずれかに分類する。

## カテゴリ定義

上位から下位への階層になっている。**ポリシーが方向を決め、設計が構造を与え、ルールが具体を縛る**。

| カテゴリ | 粒度 | 答える問い | 内容 |
| --- | --- | --- | --- |
| **ポリシー** | 上位（方向づけ） | なぜ・どこまで・どれを選ぶか | 方針・優先順位・選定・目標値・運用方針。判断の拠り所となるもの。単一の規約ではなく、方向づけや意思決定を扱う |
| **設計** | 中位（構造） | どう構成・分割するか | モジュールやコンポーネントの組み立て方・責務分割・採用パターン。構造の指針 |
| **ルール** | 下位（具体） | 具体的に何をどう書く / 設定するか | 命名・記法・スタイル・設定値など、可否が明確で機械的に判定しやすい規約 |

### 判断に迷ったときの目安

- 「**何を守るか**」が一意に決まり、守った／守っていないが判定できる → **ルール**
- 「**どう組み立てるか**」という構造・分割の話 → **設計**
- 「**なぜ・どこまで・どれを選ぶか**」という方向づけ・意思決定 → **ポリシー**

## ファイル命名規約

```
<対象><カテゴリ>[・<バリエーション>].md
```

- **対象**: テスト / コンポーネント / コーディング / ファイル命名 など、扱う領域
- **カテゴリ**: `ポリシー` / `設計` / `ルール` のいずれかを必ず付ける
- **バリエーション**（任意）: `nextjs` / `カバレッジ` など。後置し、区切りは `・`

例: `テストポリシー・nextjs.md` / `コンポーネント設計・nextjs.md` / `コーディングルール.md`
