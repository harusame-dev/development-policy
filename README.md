# development-policy

開発の方針・規約を集約するリポジトリ。
ドキュメントは**トピック単位**で1ファイルにまとめ、各ファイルの中身を **方針 → 設計 → ルール** の3層で構成する。

## 3層モデル

`方針 / 設計 / ルール` は別々の文書ではなく、**同じトピックを上位から下位へ降りていく層**。1トピックの中で、なぜ → どう組み立てるか → 具体的に何を、の順に並べる。

| 層 | 粒度 | 答える問い | 内容 |
| --- | --- | --- | --- |
| **方針** | 上位（方向づけ） | なぜ・どこまで・どれを選ぶか | 優先順位・選定・目標値・運用方針などの意思決定 |
| **設計** | 中位（構造） | どう構成・分割するか | 責務分割・採用パターンなどの構造の指針 |
| **ルール** | 下位（具体） | 具体的に何をどう書く / 設定するか | 命名・記法・設定値など、可否が明確な規約 |

すべてのトピックが3層すべてを持つわけではない。**存在する層だけ**を上記の順で記述する（薄いトピックは1層のみでよい）。

## ファイル命名規約

```
<トピック>[・<バリエーション>].md
```

- **トピック**: テスト / コンポーネント / コーディング / パッケージ管理 など、扱う領域そのもの
- カテゴリ語（`方針` / `設計` / `ルール`）はファイル名に入れない。これらは文書内の層を表す見出しとして使う
- **バリエーション**（任意）: `nextjs` / `githubactions` / `カバレッジ` など、プラットフォームや観点の差分。後置し、区切りは `・`

例: `テスト.md` / `テスト・nextjs.md` / `コンポーネント・nextjs.md`
