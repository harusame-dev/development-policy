# コーディングルール

実装時に共通して守るコーディングルールをまとめる。

## 1. 不要な抽出を避ける

### 方針

リテラル（数値・文字列）や中間変数（説明変数）の抽出は、以下のいずれかに該当する場合のみ行う。

- 2 箇所以上で参照する
- 文脈から値や式の意味が読み取れない

1 箇所でしか使わず、文脈から意味が自明な値はインラインで書く。

### 理由

- **読む際の追跡コストを減らす**: 抽出された変数は、参照箇所から定義箇所へ視線を往復させる必要がある。意味が自明な値を抽出すると、得られる情報量に対して追跡コストだけが増える
- **抽出は重複か命名のためにある**: 同じ値を複数箇所で使う場合は一元管理のために、意味が読み取りにくい値は名前で意図を示すために抽出する。この目的が無い抽出は、コードの行数を増やすだけで価値を生まない

### 実装方法

```tsx
// 非推奨: 1 箇所でしか使わず、意味も自明な値を抽出している
const ZERO = 0;
const isEmpty = items.length === ZERO;

// 推奨: 自明な値はインラインで書く
const isEmpty = items.length === 0;
```

```tsx
// 推奨: 文脈から意味が読み取れない値は説明変数で意図を示す
const isAdult = age >= 18;

// 推奨: 2 箇所以上で参照する値は定数に抽出して一元管理する
const MAX_RETRY_COUNT = 3;
```

## 2. 文字列リテラル

### 方針

2 箇所以上で参照される文字列リテラルの集合（ErrorCode・ステータス・種別・discriminated union のタグなど）は、次のように定義する。

- `as const` Object と、その値型 `(typeof X)[keyof typeof X]` を**同名**で定義する
- 変数名・メンバ名は **PascalCase**
- DB / 外部 API の値を表す場合は、その値をそのまま使用する
- 内部の値はキーと同じ PascalCase に揃える

### 理由

- **値と型を一対で扱える**: 同名で `as const` Object と値型を定義することで、実行時に使う値（`ErrorCode.NotFound`）と型注釈（`ErrorCode`）を 1 つの import で揃えて使える
- **値の追加に型が自動追従する**: `(typeof X)[keyof typeof X]` で Union を導出するため、メンバを足すだけで型側も追従し、Union を二重管理しなくて済む
- **境界では外部の値に合わせる**: DB / 外部 API が持つ値をそのまま採用すれば、永続化・通信の境界で値の変換が不要になり、保存値と一致する
- **内部値はキーに揃えて一貫させる**: 外部由来でない値はキーと同じ PascalCase にすることで、キーと値の対応に迷いがなくなり、表記が一貫する

### 実装方法

```ts
// 内部の値: キーと同じ PascalCase に揃える
export const ErrorCode = {
  NotFound: "NotFound",
  Unauthorized: "Unauthorized",
} as const;
export type ErrorCode = (typeof ErrorCode)[keyof typeof ErrorCode];
// type ErrorCode = "NotFound" | "Unauthorized"
```

```ts
// DB / 外部 API の値: その値をそのまま使用する
export const UserStatus = {
  Active: "active", // DB に保存される値
  Suspended: "suspended",
} as const;
export type UserStatus = (typeof UserStatus)[keyof typeof UserStatus];
// type UserStatus = "active" | "suspended"
```
