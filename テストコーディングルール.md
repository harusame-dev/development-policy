# テストコーディングルール

テストコードを書くうえでのルールをまとめる。テスト全体の方針は [テストポリシー](./テストポリシー.md) を参照する。

## フロントエンド・テスト

### 方針

- `it` は使わず、`test` を使う
- `describe` は使用しない
- データの投入などのセットアップ・ティアダウンは `test.extend`（fixture）を使用する
- テスト用ユーティリティ（`lib/test/**` 等の fixture / ヘルパー）に対するテストファイルは作成しない

### 理由

- **記法を統一して揺れをなくす**: `it` / `test` はどちらも同じ機能だが、混在すると検索性・一貫性が下がる。`test` に統一する
- **構造はファイルで表し、ネストを増やさない**: `describe` でのグルーピングはネストとインデントを深くする。テストはファイル分割で構造化し、各 `test` をフラットに保つことで見通しを良くする
- **fixture で依存を明示し、並列実行に強くする**: `beforeEach` / `afterEach` による暗黙の共有状態ではなく、`test.extend` で各テストに必要なものを引数として明示的に注入する。依存が読み取りやすく、テスト間の状態共有による不安定さも避けられる
- **テスト基盤自体はテストしない**: `lib/test/**` のヘルパーは利用側テストの中で実質的に動作確認される。ここにテストを書くと、テストのためのテストが増え保守コストだけがかさむ

### 実装方法

セットアップ・ティアダウンは `test.extend` で fixture として定義し、`use` の前後で行う。

```ts
import { test as base } from "vitest";

const test = base.extend<{ user: User }>({
  user: async ({}, use) => {
    const user = await createUser(); // セットアップ
    await use(user);
    await deleteUser(user.id); // ティアダウン
  },
});

// describe で囲まず、test をフラットに並べる
test("ユーザー名を表示する", ({ user }) => {
  // user を使ったテスト
});
```
