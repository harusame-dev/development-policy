# ファイル命名

フレームワークや言語を問わず、すべてのファイル命名で共通して守るルールをまとめる。フレームワークや言語固有の命名（サフィックスの具体パターンなど）は、本ファイルを前提としたうえで各バリエーション（`ファイル命名・<バリエーション>.md`）に記載する。

## 1. ケバブケース

### 方針

ファイル名は原則としてケバブケース（小文字 + ハイフン区切り、例: `user-profile`）で命名する。

ただし、フレームワークやツールが予約するファイル名（Next.js の `page.tsx` / `layout.tsx`、`README.md` など）は対象外とし、その規約に従う。

### 理由

- **環境差による事故を防ぐ**: 大文字小文字を区別するファイルシステム（Linux）と区別しないファイルシステム（macOS / Windows）が混在すると、`UserProfile.ts` と `userprofile.ts` のような表記揺れが環境によって別物・同一物に分かれ、import 切れや Git 上の二重管理を招く。小文字に統一することでこの差異を踏まない
- **表記を一意に保てる**: camelCase / PascalCase / snake_case が混在すると、同じ概念のファイルでも命名が人によってぶれる。ケバブケースに固定することで、ファイル名の付け方に迷いがなくなり一貫する

## 2. サフィックスで種類を表す

### 方針

ファイルの種類・役割は、ファイル名のサフィックス（拡張子の前に付ける `.<種類>`、例: `*.test.ts` / `*.config.ts`）で表現する。

どの種類にどのサフィックスを使うかは、フレームワーク・言語によって異なるため各バリエーションで定義する。

### 理由

- **ファイル名だけで役割が分かる**: サフィックスで種類を表現することで、AI も人も中身を開かずにファイル名だけでそのファイルが何かを判別できる。AI に編集・生成を任せる際、誤った種別での実装や取り違えを減らせる
- **rule の `paths` で必要なコンテキストを最小限・確実に読み込める**: 命名が規則的だと、AI 向けの rule ファイルのフロントマター `paths` で対象ファイル群を glob 指定できる。これにより、対象ファイルに対して必要なルール／コンテキストだけを最小限かつ確実に読み込ませられ、無関係な情報でコンテキストを汚さずに済む
