# パッケージ管理のガイドライン

## 概要

パッケージ管理におけるセキュリティ・品質・メンテナンス性を統一するためのガイドライン。

## 1. インストール可能なパッケージの公開日数

### 方針

パッケージ公開から 14 日以上経過していないパッケージのインストールを禁止する。

### 理由

- 悪意のあるパッケージは通常数時間以内に検知・削除されるため、インストールを遅らせるだけで、コンパイル時に汚染版を踏むリスクを大幅に下げられる
- 14 日という数字は Renovate で automerge をする場合に推奨されている値で、レジストリが悪意のある依存関係を削除する時間を十分に確保できる
- メジャーアップデート以外は基本的に automerge 前提のため、automerge 推奨値である 14 日に揃える

### 実装方法

#### pnpm

`pnpm-workspace.yaml` で `minimumReleaseAge` を設定する（pnpm 11 以降、`pnpm-workspace.yaml` が正式な設定箇所）。

```yaml
# pnpm-workspace.yaml
minimumReleaseAge: 20160
```

`minimumReleaseAge` の単位は分のため、14 日 = `20160` 分を指定する。

#### Renovate

`renovate.json` に `minimumReleaseAge` を設定する。

```json
{
  "extends": ["config:recommended"],
  "minimumReleaseAge": "14 days"
}
```

#### Go

Go には `minimumReleaseAge` に相当する機能が存在しないため、CI で `gomod-age` によるチェックワークフローを必ず作成する。

ただし、ローカルでの `go get` などによるインストールを CI では防げない。根本的な対策が打てないため、新しいパッケージを導入する際はリリースから 14 日以上経過しているバージョンを指定すること。

## 2. JavaScript パッケージのバージョン指定

### 方針

`package.json` ではバージョン範囲指定（`^1.2.3` や `~1.2.3`）ではなく、exact version（`1.2.3`）でインストールすること。

### 理由

Renovate が推奨しており、以下のメリットがあるため。

- **何が動いているかが明確になる**: `package.json` に固定バージョンが書かれていれば、誰がいつインストールしても同じバージョンが入る。範囲指定の場合は、インストール時期やロックファイルの状態によって実際に入るバージョンが変わるため、何が動いているかが分かりづらくなる
- **障害発生時の切り分けが容易になる**: 不具合のあるバージョンがリリースされた場合、範囲指定だと知らないうちに本番に取り込まれて原因特定が難しくなる。exact version であれば、バージョンを上げた PR が CI で失敗し、その PR をマージしない判断ができるため、`main` ブランチや本番環境を壊さずに済む
- **更新が PR として明示される**: バージョンアップは必ず Renovate の PR として現れるため、変更内容・タイミング・レビュー履歴が残り、変更管理が一貫する

### 実装方法

#### pnpm

`pnpm-workspace.yaml` で `savePrefix` を空文字に設定する（pnpm 11 以降、`pnpm-workspace.yaml` が正式な設定箇所）。

```yaml
# pnpm-workspace.yaml
savePrefix: ""
```

`savePrefix` のデフォルトは `^`。空文字を指定することで `pnpm add` 実行時に `^` や `~` が付かない exact version で `package.json` に記録される。

#### Renovate

`renovate.json` で `config:pinAllExceptPeerDependencies` を有効化する。

```json
{
  "extends": ["config:recommended", "config:pinAllExceptPeerDependencies"]
}
```

`peerDependencies` 以外のすべての依存関係を exact version にピン留めする。既存の範囲指定もピン留めする PR が自動で作成される。

## 3. パッケージの自動更新

### 方針

パッケージの自動更新を必ず導入すること。

### 理由

- **セキュリティ脆弱性への追従コストを下げる**: 脆弱性が公表されてから手動で更新版を確認・適用するのは追従漏れが発生しやすい。自動化により脆弱性修正版が PR 化されるため、対応のリードタイムを短くできる
- **手動更新の運用負荷を下げる**: 依存関係が増えるほど手動更新は現実的でなくなる。検知・PR 作成までが自動化されるため、レビューとマージに集中できる
- **依存関係の鮮度を保てる**: 更新が後回しになると一度に大量のメジャーアップデートを処理することになり、破壊的変更の影響範囲が広がる。継続的に小さく更新することで、変更の影響を局所化できる

### 実装方法

[Renovate](https://docs.renovatebot.com/) を導入する。
