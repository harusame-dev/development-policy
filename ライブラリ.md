# ライブラリ

ライブラリ・外部パッケージの選定方針と、利用時の設計・ルールをまとめる。

## 1. ライブラリ選定

### 方針

利用対象ごとに、原則として以下のライブラリを採用する。

| 利用対象 | ライブラリ | 理由 |
| --- | --- | --- |
| UI コンポーネント | shadcn/ui | デフォルトでアクセシビリティ対応かつ質の高いデザインが用意されており、さらにコードをフルカスタマイズできるため細かい要件にも対応できる |
| 日付操作 | date-fns | 関数単位でインポートでき、バンドルサイズを小さく保てる |
| バリデーション | valibot | モジュラー API を当初から設計思想として作り込んでおり成熟度・一貫性が高く、必要な機能だけを取り込めてバンドルサイズも小さく保てる（同種の zod-mini は後発の変種） |
| フォーマット | Prettier | デファクトスタンダードで設定の余地が少なく、整形を一意化できる |
| 静的解析（Lint） | ESLint | プラグイン・ルールのエコシステムが最も広く、規約の強制やラッパー化の徹底（`no-restricted-imports`）に対応できる |
| デッドコード検出 | Knip | 未使用の export・依存・ファイルを横断的に検出でき、モノレポにも対応する |
| テスト | Vitest | 高速で、フィクスチャ（`test.extend`）・カバレッジを標準で備える |
| UI テスト | Vitest（Browser Mode） | 実ブラウザ上でコンポーネントを描画・操作でき、テストランナーを Vitest に統一できる |
| E2E テスト | Playwright | クロスブラウザ対応で、アクセシビリティクエリによる要素操作・フィクスチャを備える |
| アクセシビリティチェック | Playwright（@axe-core/playwright） | E2E と同一基盤で実行時のアクセシビリティ検証を組み込める |

## 2. 外部パッケージのラッパー化

### 方針

外部パッケージは呼び出し側から直接 import せず、**アプリのユースケースに限定した薄いラッパー**を 1 箇所に置き、アプリ側はそのラッパー経由でのみ利用する。

ラッパーの対象は、特定のユースケースのために利用する差し替え可能なサードパーティ製ライブラリ（日付操作・バリデーション・HTTP クライアント・ロガー・決済 SDK など）。言語ランタイムやフレームワーク（React / Next.js など）、リポジトリにコードを取り込んで自分のコードとして管理するもの（shadcn/ui など）は対象外。

### 理由

- **差し替え時の影響範囲を閉じ込められる**: ライブラリを別のものに置き換える際、修正対象がラッパー内に限定される。呼び出し側はラッパーの API に依存しているため変更不要で、影響範囲の見落としが起きない
- **更新の自動化を裏付けられる**: ユースケースに限定したラッパーには単体テストを書ける。ライブラリ更新時にそのテストが通れば「プロジェクトの利用方法に影響がない」ことを機械的に確認でき、automerge による更新の自動化を安全に回せる
- **依存を必要十分に保てる**: アプリが必要とする最小の API だけを公開するため、ライブラリの広い API 表面に呼び出し側が引きずられず、実際に使う機能だけに依存を絞れる

### 設計

- **ラッパーはアプリのユースケース語彙で API を定義する**: ライブラリの関数名・引数体系をそのまま横流しせず、アプリがやりたいこと（例: 「表示用に日付を整形する」）を表す名前と最小の引数で公開する
- **ライブラリ固有の型を境界から漏らさない**: ラッパーの引数・戻り値はアプリ側の型かプリミティブに限定し、ライブラリ固有の型を外に出さない。これにより差し替え時の影響がラッパー内で完結する
- **1 ライブラリにつき 1 ラッパーモジュール**: 配置場所を統一し（例: `lib/<対象>.ts`）、そのライブラリの import がリポジトリ内で 1 箇所に集約される状態を保つ

### ルール

- ラッパーモジュール以外からの当該ライブラリの直接 import を禁止する。ESLint の `no-restricted-imports` で強制し、ラッパーモジュール自身のみ例外として許可する
- ラッパーには単体テストを必須とする。公開する各関数のユースケースを網羅し、ライブラリ更新時の回帰検知の土台にする
- 現に使うユースケースだけをラップする。将来使うかもしれない機能を先回りで公開しない（公開した分だけテストと差し替えコストが増える）

### 実装方法

ユースケースに限定したラッパーを 1 箇所に定義する。

```ts
// lib/date.ts — date-fns のラッパー。アプリで使う日付ユースケースだけを公開する
import { format, addDays, isBefore } from "date-fns";

/** 表示用に日付を YYYY-MM-DD へ整形する */
export function formatYmd(date: Date): string {
  return format(date, "yyyy-MM-dd");
}

/** 指定日数後の日付を返す */
export function addDaysFrom(date: Date, days: number): Date {
  return addDays(date, days);
}

/** date が基準時刻より過去かを判定する */
export function isPast(date: Date, now: Date): boolean {
  return isBefore(date, now);
}
```

呼び出し側はライブラリではなくラッパーを参照する。

```ts
// 非推奨: 呼び出し側がライブラリへ直接依存している
import { format } from "date-fns";
const label = format(date, "yyyy-MM-dd");

// 推奨: ユースケースを表すラッパー経由で利用する
import { formatYmd } from "@/lib/date";
const label = formatYmd(date);
```

ラッパーには公開関数ごとに単体テストを用意し、ライブラリ更新時の回帰検知の土台にする。

```ts
// lib/date.test.ts
import { expect, test } from "vitest";
import { formatYmd, isPast } from "./date";

test("formatYmd は YYYY-MM-DD で整形する", () => {
  expect(formatYmd(new Date("2026-06-13T10:00:00Z"))).toBe("2026-06-13");
});

test("isPast は基準時刻より前なら true", () => {
  const now = new Date("2026-06-13T00:00:00Z");
  expect(isPast(new Date("2026-06-12T00:00:00Z"), now)).toBe(true);
});
```

ESLint で直接 import を禁止し、ラッパーモジュールのみ例外として許可する。

```js
// eslint.config.js
import { defineConfig } from "eslint/config";

export default defineConfig([
  {
    rules: {
      "no-restricted-imports": [
        "error",
        {
          paths: [
            {
              name: "date-fns",
              message: "lib/date のラッパー経由で利用してください。",
            },
          ],
        },
      ],
    },
  },
  {
    // ラッパーモジュール自身はライブラリを直接 import してよい
    files: ["lib/date.ts"],
    rules: {
      "no-restricted-imports": "off",
    },
  },
]);
```
