# 環境変数のガイドライン

## 概要

環境変数の参照方法を統一し、タイポ・未設定・形式不正といった不備を実行時ではなくビルド時に検出するためのガイドライン。

## 1. process.env の直接参照の禁止

### 方針

アプリコードで `process.env` を直接参照することを禁止し、専用の設定モジュール（`public-config.ts` / `private-config.ts`）からのみ参照する。

- `public-config.ts`：クライアントに公開してよい変数（Next.js では `NEXT_PUBLIC_` プレフィックス必須）
- `private-config.ts`：サーバー専用の秘密情報（`NEXT_PUBLIC_` プレフィックス禁止）

### 理由

- `process.env` は型定義が効かないため、環境変数名のタイポにコンパイル時に気づけない
- 参照のたびに undefined チェックやパース（数値変換など）を繰り返す必要があり、漏れが起きやすい
- 公開可能な変数と秘密情報をファイルで分離することで、秘密情報のクライアントへの露出を仕組みで防ぐ

## 2. 環境変数の不備のビルド時検出

### 方針

スキーマのパースは設定モジュールのトップレベル（グローバル）で実行し、環境変数の不備をビルド時に検出する。

### 理由

- 関数内でパースすると、その関数が実際に呼ばれる実行時まで不備に気づけない
- トップレベルで実行すれば、設定モジュールが import された時点（= ビルド時）で失敗し、デプロイ前に確実に検出できる

## 3. スキーマ定義

### 方針

環境変数のスキーマは valibot で定義する。

### 理由

- パース結果に型が付き、参照側で型安全に扱える
- 未設定チェックに加えて形式チェック（URL 形式など）を宣言的に記述できる
- tree-shaking が効きバンドルサイズが小さいため、クライアントに載る `public-config.ts` でも使える

### 実装例

```ts
// public-config.ts
import * as v from "valibot";

const publicConfigSchema = v.object({
  url: v.pipe(
    v.string("NEXT_PUBLIC_SUPABASE_URL を設定してください"),
    v.url("NEXT_PUBLIC_SUPABASE_URL は URL 形式で設定してください"),
  ),
  publicKey: v.pipe(
    v.string("NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY を設定してください"),
    v.nonEmpty("NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY を設定してください"),
  ),
});

// モジュール読み込み時にパースし、環境変数の不備をビルド時点で検出する
const publicConfig = v.parse(publicConfigSchema, {
  url: process.env.NEXT_PUBLIC_SUPABASE_URL,
  publicKey: process.env.NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY,
});

export function getSupabasePublicConfig() {
  return publicConfig;
}
```

## 4. Next.js 固有の注意点

### 方針

`NEXT_PUBLIC_` 変数は `process.env.NEXT_PUBLIC_X` の形の静的参照を維持し、`process.env[name]` のような動的参照を禁止する。

### 理由

Next.js は `NEXT_PUBLIC_` 変数をビルド時に静的参照箇所へインライン展開するため、動的参照にするとクライアントバンドルで値が取得できなくなる。
